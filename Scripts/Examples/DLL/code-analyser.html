Project summary

Что это за проект по структуре и стилю

Это небольшой демонстрационный проект Windows DLL, ориентированный на вызов из MQL4-клиента. По структуре он предельно простой: основная логика сосредоточена практически в одном исходном файле DLL, рядом лежит проект Visual C++ и тестовый скрипт/клиент для MetaTrader. Архитектурно это процедурный API для обмена простыми типами, строками, массивами и структурами между DLL и внешней средой.

Стиль реализации выглядит как учебный или экспериментальный: минимум слоев, почти нет формализованного интерфейса, контракт DLL задается в основном самой реализацией, а не отдельным публичным заголовком. При этом видны попытки учесть бинарную совместимость — используются экспорт функций, фиксированное соглашение вызова и упаковка структуры, — но эти решения не доведены до единого и строгого ABI-контракта.

  • Нет явного центрального интерфейсного заголовка, который был бы единым источником истины для клиента и DLL.
  • API процедурный, “плоский”, без явной модели версионирования и без четкой системы ошибок.
  • На границе DLL активно используются сырые указатели и внешние буферы, что делает пример наглядным, но хрупким.
  • Стиль кода местами непоследовательный: смешиваются разные типы флагов успеха и варианты calling convention.

Главные архитектурные проблемы

Ключевая архитектурная проблема проекта — отсутствие формально описанного DLL-контракта. Сейчас совместимость между DLL и клиентом держится на неявных договоренностях: как упакована структура, какой тип реально ожидается, как трактуются строки, что означает возвращаемое значение при ошибке. Для бинарного интерфейса это слишком слабая основа.

  • Нет общего экспортного заголовка с extern "C", единым макросом экспорта, calling convention, packing и описанием структур.
  • Модель API несогласованна: часть функций возвращает результат напрямую, часть — флаг успеха, часть — длину или количество.
  • Ошибки не унифицированы: нет общей схемы кодов ошибок и нет явного отделения “ошибка” от “корректное нулевое значение”.
  • Контракт на строки и буферы недостаточно формализован: неочевидно, кто владеет памятью, обязателен ли нуль-терминатор и как интерпретируется длина.
  • Структура данных и функции доступа выглядят несинхронно: публично раскрывается не весь смысл структуры, что делает API неполным и местами случайным.
  • Выбор “сырого” интерфейса на указателях и массивах повышает риск ABI-расхождений между компилятором DLL и внешним клиентом.

Самые опасные баги и риски

Самые серьезные риски сосредоточены на границе DLL: именно там нестрогий контракт и небезопасные операции быстро превращаются в повреждение памяти, выход за границы буфера или трудноуловимые проблемы совместимости. Для такого проекта это важнее, чем локальные алгоритмические недочеты.

  • Небезопасная работа со строками: использование strlen и strcat на внешних буферах без достаточных ограничений и проверок.
  • Возможны чтение за пределами памяти и сбои, если входная строка не нуль-терминирована в пределах переданного буфера.
  • Есть риск переполнения буфера на 1 байт в логике обработки массивов строк из-за некорректной проверки места под завершающий ноль.
  • Использование неверного формата для wide-строк в выводе может приводить к некорректной диагностике или неопределенному поведению.
  • Возврат 64-битных значений через double создает потерю точности и искажает публичный контракт.
  • Возврат 0.0 как сигнала ошибки опасен, потому что это может быть и валидное значение.
  • Некорректный или сомнительный компаратор для qsort может приводить к неопределенному поведению при сортировке.
  • Отсутствие общего ABI-описания создает риск несовпадения типов, упаковки структур и соглашения вызова между DLL и MQL4-клиентом.

Проблемы поддерживаемости и читаемости

Проект маленький, но поддерживать его в текущем виде будет сложно именно из-за неформализованности. Когда интерфейс живет только в .cpp-файле, любое изменение требует ручной синхронизации знаний между кодом DLL и внешним клиентом. Это повышает цену даже простых доработок.

  • Отсутствует явный публичный header-файл, поэтому контракт трудно читать, проверять и переиспользовать.
  • Есть непоследовательность в типах: одновременно встречаются bool и BOOL.
  • Используются разные записи calling convention: __stdcall и _stdcall, что ухудшает единообразие.
  • Система возвратов и ошибок неоднородна, из-за чего поведение функций приходится угадывать по месту.
  • Есть признаки копипаста: неточные сообщения логирования и несоответствие имен функций в диагностике.
  • Недостаточно самодокументируемых комментариев именно по ABI-правилам: упаковка, размеры, строки, массивы, допустимые значения параметров.
  • API скорее демонстрирует отдельные приемы, чем представляет цельную библиотеку с ясными границами ответственности.

Быстрые улучшения

Без большой переработки архитектуры уже можно заметно снизить риски и сделать проект понятнее. Наибольший эффект дадут изменения, которые одновременно улучшают безопасность и формализуют поведение функций.

  • Добавить единый экспортный заголовок с объявлениями всех функций, структур, packing, extern "C" и единым макросом экспорта.
  • Унифицировать соглашение вызова и типы на DLL-границе: выбрать один стиль и использовать его везде.
  • Перевести API на схему BOOL/int + out-параметры там, где сейчас ошибка маскируется обычным возвращаемым значением.
  • Для строк ввести жесткие правила: обязательная проверка NULL, явная длина буфера, требование места под '\0', безопасные bounded-операции.
  • Исправить форматирование wide-строк и прочие локальные дефекты, влияющие на корректность и диагностику.
  • Убрать возврат 64-битных значений через double, заменить на корректные типы или отдельные выходные параметры.
  • Исправить компаратор сортировки, проверки размеров и граничные условия для массивов и строк.
  • Привести сообщения логирования к точным именам функций и единым формулировкам.

Приоритетный план доработок

Оптимальный порядок работ — сначала устранить риски повреждения памяти и неоднозначности ABI, затем выровнять API и только после этого заниматься косметикой и расширением функциональности.

  • Шаг 1: ввести публичный ABI-заголовок с полным описанием структур, packing, экспорта, calling convention и правил использования строк/массивов.
  • Шаг 2: срочно переписать небезопасные строковые участки на ограниченные операции с четкими проверками размеров и нуль-терминации.
  • Шаг 3: унифицировать модель возврата результатов и ошибок; исключить случаи, где валидное значение одновременно означает ошибку.
  • Шаг 4: исправить публичный контракт для 64-битных данных, не использовать double как транспорт для целых значений.
  • Шаг 5: стандартизировать типы на границе DLL, убрать смешение BOOL/bool и разнобой в calling convention.
  • Шаг 6: закрыть локальные дефекты высокой вероятности: off-by-one в буферах, компаратор qsort, проверки NULL, спорные граничные условия.
  • Шаг 7: улучшить читаемость и сопровождение — вычистить логирование, добавить комментарии к ABI и, при необходимости, разбить реализацию по тематическим модулям.

Назначение класса/модуля

Модуль представляет собой демонстрационную WinAPI DLL для вызова из MQL4. Экспортирует набор простых процедур для передачи через DLL границу скаляров, строк, массивов чисел, массива котировок и массива строковых структур. Реализация полностью procedural, заголовочный файл отсутствует, контракт фактически задается только экспортируемыми сигнатурами в DLLSample.cpp:54, DLLSample.cpp:62, DLLSample.cpp:70, DLLSample.cpp:78, DLLSample.cpp:102, DLLSample.cpp:127, DLLSample.cpp:170, DLLSample.cpp:191.

Что видно по контракту и структуре

  • Внешний ABI формируется через __declspec(dllexport) и __stdcall; это важно для совместимости с MQL4, см. DLLSample.cpp:10, DLLSample.cpp:54.
  • Структура RateInfo упакована в 1 байт, что явно фиксирует layout на DLL-границе, см. DLLSample.cpp:14–DLLSample.cpp:26.
  • MqlStr содержит длину буфера и указатель на ANSI-строку, но фактический контракт длины не документирован: неясно, включает ли len место под завершающий ноль, см. DLLSample.cpp:28–DLLSample.cpp:32.
  • Ошибки возвращаются через значения функций, исключений нет; это соответствует ограничениям проекта. Однако коды ошибок не унифицированы: где-то возвращается 0.0, где-то -1, где-то FALSE, см. DLLSample.cpp:81–DLLSample.cpp:95, DLLSample.cpp:173–DLLSample.cpp:181.
  • Модуль использует CRT I/O и строки (printf, wprintf, strlen, strcmp, strcat, qsort), что для DLL допустимо технически, но контракт памяти и потокобезопасности с внешним рантаймом не описан, см. DLLSample.cpp:56, DLLSample.cpp:72, DLLSample.cpp:184, DLLSample.cpp:211, DLLSample.cpp:227.

Замечания по качеству кода

  • У модуля нет отдельного заголовка, поэтому экспортируемый контракт, packing структур и соглашение о вызовах нельзя переиспользовать из одного источника истины. Это повышает риск расхождения между DLL и клиентом, см. DLLSample.cpp:14, DLLSample.cpp:28, DLLSample.cpp:54.
  • Есть непоследовательность в стиле вызова: используется и __stdcall, и _stdcall. Для ABI это обычно эквивалентно, но для читаемости и единообразия лучше выбрать один вариант, см. DLLSample.cpp:54, DLLSample.cpp:102.
  • Используются C-style cast и снятие const в компараторе, хотя модификация объектов не требуется, см. DLLSample.cpp:237–DLLSample.cpp:238.
  • Возврат 0.0 как ошибки в числовых функциях двусмысленен, потому что это может быть и валидное значение данных, см. DLLSample.cpp:83–DLLSample.cpp:95, DLLSample.cpp:132–DLLSample.cpp:165.
  • DllMain сейчас безопасно пустой, что хорошо; видимых проблем внутри него нет, см. DLLSample.cpp:37–DLLSample.cpp:50.

Архитектурные замечания

  • Для DLL-границы смешаны несколько моделей контракта: прямой возврат значения, возврат флага успеха, возврат длины массива. Для внешнего клиента надежнее единая схема: BOOL/int result + out-параметры, см. DLLSample.cpp:54–DLLSample.cpp:57, DLLSample.cpp:78–DLLSample.cpp:98, DLLSample.cpp:170–DLLSample.cpp:187.
  • Контракт строк через char*/wchar_t* недостаточно формализован: кто владеет памятью, гарантируется ли нуль-терминатор, что означает len, допустимы ли NULL-строки. Эти правила сейчас размазаны по условиям в коде, но не выражены интерфейсом, см. DLLSample.cpp:28–DLLSample.cpp:32, DLLSample.cpp:206–DLLSample.cpp:227.
  • RateInfo уже содержит больше полей, чем реально доступны через GetRatesItemValue: не экспортируются spread и vol_real. Это делает ABI неполным и намекает на несогласованный дизайн между структурой и API, см. DLLSample.cpp:23–DLLSample.cpp:24, DLLSample.cpp:148–DLLSample.cpp:163.
